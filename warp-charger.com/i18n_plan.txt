i18n Plan (de + en, URL prefix /de/... and /en/...)
====================================================

Current state: All templates are German-only, routes have no language prefix.

1. Template generalization (data-driven i18n)
   - Templates stay as shared HTML in templates/sections/ and templates/pages/
     (no duplication into de/ and en/ dirs)
   - Extract all translatable text into a data structure, e.g. translations/ dir
     with YAML or Python dicts per section/page:
       translations/de/hero.yaml   -> title, subtitle, cta_label, price_prefix...
       translations/en/hero.yaml   -> title, subtitle, cta_label, price_prefix...
   - Templates use variables: {{ t.title }}, {{ t.cta_label }}, etc.
   - To add a new language: only add a new translations/<lang>/ file set
   - base.html receives {{ lang }} variable, sets <html lang="{{ lang }}">

2. Translated URL slugs + cross-language redirects
   - Define PAGES dict with canonical slugs per language:
       de: produkte/warp4-charger
       en: products/warp4-charger
   - Define SLUG_ALTERNATES mapping between languages:
       ("de", "produkte/warp4-charger") <-> ("en", "products/warp4-charger")
   - Cross-language redirect rule: if a user visits /<lang>/<slug> where <slug>
     belongs to a different language, 301-redirect to /<lang>/<canonical_slug>.
     Example: /en/produkte/warp4-charger -> 301 -> /en/products/warp4-charger
     This means users can always just swap "de" <-> "en" in the URL and it works.

3. Routes
   - /                        -> detect Accept-Language, redirect to /de/ or /en/
   - /<lang>/                 -> homepage (render sections with translations/<lang>/)
   - /<lang>/<path:slug>      -> sub-pages; look up in PAGES[lang], if not found
                                  check if slug exists in other lang -> 301 redirect
                                  to correct slug for this lang. Otherwise 404.
   - /section/<lang>/<name>   -> single section preview (dev only)
   - Unknown lang prefix -> 404

4. Language-aware partials (header, footer, mobile_menu)
   - Nav links use {{ lang }} prefix and translated slugs from PAGES[lang]
   - Link text comes from translations/<lang>/nav.yaml (or similar)
   - External links (docs, shop, social) stay unchanged

5. Language switcher
   - DE | EN toggle in header (desktop + mobile)
   - Uses SLUG_ALTERNATES to find the equivalent page in the other language
   - Pass alternate URL to templates for the switcher link

6. SEO
   - <link rel="alternate" hreflang="de" href="..."> and hreflang="en" in <head>
   - Use SLUG_ALTERNATES for correct canonical URLs
   - 301 redirects for cross-language slugs preserve link juice

7. Sitemap.xml
   - Generate sitemap with entries for both languages
   - Each <url> includes <xhtml:link rel="alternate" hreflang="de" href="...">
     and hreflang="en" counterpart
   - Serve at /sitemap.xml (can be a dynamic route or generated at build time)

8. Localized 404 page
   - /de/... routes show German 404 text, /en/... show English 404 text
   - 404 translations go into translations/<lang>/404.yaml
   - Reuse base layout so header/footer remain language-aware

9. Translated meta tags
   - <title>, <meta name="description">, og:title, og:description need translation
   - Add per-page meta fields to translations/<lang>/<page>.yaml:
       meta_title, meta_description, og_title, og_description
   - base.html renders these from {{ t.meta_title }}, etc.

10. Legacy URL redirects (migration)
    - Current site serves pages without language prefix (e.g. /produkte/warp4-charger)
    - On deploy, add 301 redirects from old prefix-less URLs to /de/<slug>
    - Implement as a catch-all route: if path has no lang prefix and matches
      a known German slug, redirect to /de/<slug>
    - Preserves existing SEO rankings and inbound links
